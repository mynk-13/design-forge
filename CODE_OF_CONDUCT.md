# Code of Conduct

## Our Pledge

DesignForge is an open-source React component design system built to be genuinely useful — for developers integrating it into their projects and for contributors who improve it. We are committed to making participation in this project a respectful, harassment-free experience for everyone, regardless of age, body size, visible or invisible disability, ethnicity, sex characteristics, gender identity and expression, level of experience, education, socio-economic status, nationality, personal appearance, race, caste, color, religion, or sexual identity and orientation.

Whether you are opening your first issue, reviewing a component API proposal, contributing a bug fix, or debating the merits of a design token — you are welcome here.

---

## Our Standards

### Behavior we encourage

- **Constructive technical discourse** — When disagreeing about an API shape, accessibility approach, or component design decision, explain your reasoning. "This variant naming breaks the existing pattern because..." is far more useful than "this is wrong."
- **Accessibility-first thinking** — DesignForge ships with axe-core zero-violation tests and WCAG 2.1 AA compliance as a baseline. Contributions that strengthen this are valued; pushback against accessibility requirements without justification is not.
- **Respectful code reviews** — Focus on the code, not the contributor. Suggest, don't demand. If a PR approach has merit but needs changes, acknowledge the effort before requesting revisions.
- **Inclusive component design** — When proposing new components or APIs, consider how they will behave across screen readers, keyboard navigation, reduced-motion preferences, and high-contrast modes — not just the happy path.
- **Welcoming newcomers** — If someone asks a question that seems basic, answer it. Everyone was new to React, TypeScript, or design systems at some point.
- **Honest, proportionate feedback** — Specific is better than vague. "The `variant` prop should be a union type, not a string" is a good review comment. "The whole approach is bad" is not.

### Behavior that is not acceptable

- Harassment, trolling, or sustained personal attacks in issues, PRs, or discussions
- Condescension about technology choices (e.g., dismissing Tailwind CSS, Radix UI, or TypeScript strict mode as invalid approaches)
- Gatekeeping participation based on experience level — contributions from new developers are as welcome as those from senior engineers, provided they meet the project's quality bar
- Dismissing accessibility requirements as optional or "nice to have"
- Publishing private information about other contributors (email addresses, personal details) without consent
- Using project spaces (issues, discussions, PR comments) for off-topic promotion or spam
- Any conduct that would be considered unprofessional or inappropriate in a collaborative software environment

---

## Scope

This Code of Conduct applies to all community spaces associated with the DesignForge project:

- GitHub repository: issues, pull requests, code review comments, discussions, and commit messages
- The `@designforge/*` npm package pages (package descriptions, README content)
- Any official communication made on behalf of the project (social media posts, blog posts, conference presentations)

It applies to all contributors — whether opening a one-line typo fix or proposing a new component package.

## Enforcement Guidelines

Consequences for violations are proportional to impact:

### 1. Correction

**Impact:** A single instance of inappropriate language or unprofessional conduct.

**Consequence:** A private message from the maintainer explaining what happened and why it was a problem. A public apology may be requested.

### 2. Warning

**Impact:** A violation or pattern of minor violations that continues after a correction.

**Consequence:** A formal warning. The contributor must not interact with the people involved (or anyone enforcing this Code) for a defined period. Continued violations lead to a temporary ban.

### 3. Temporary Ban

**Impact:** Serious or sustained violations — including harassment, repeated dismissal of accessibility requirements as a project value, or sustained hostile behavior in reviews or discussions.

**Consequence:** A temporary ban from all project spaces (repository interactions, discussions). Duration is set by the maintainer based on severity.

### 4. Permanent Ban

**Impact:** A pattern of violations, harassment targeting specific individuals, or extreme conduct that undermines the project community.

**Consequence:** Permanent removal from all project spaces with no path to reinstatement.
